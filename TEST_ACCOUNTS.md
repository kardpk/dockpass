# BoatCheckin Testing & Visual Verification Guide

To perform visual verifications across the platform, you need to understand how each of the three primary actors accesses the system. Unlike typical applications where everyone uses the same login page, BoatCheckin heavily leverages **stateless links** for Guests and Captains to eliminate friction.

---

## 1. The Operator View (Dashboard)
The Operator is the business owner who manages boats, trips, and settings. This is the only role that uses a traditional email and password login.

* **URL to Test**: `http://localhost:3000/login`
* **Test Credentials**:
  - **Email**: `operator@dockpass.test`
  - **Password**: `password123`
*(Note: If you haven't created this account yet, go to `http://localhost:3000/signup` and register with these exact details).*

**Key Routes to Verify as Operator:**
- `/dashboard/trips` (Creating and managing trips)
- `/dashboard/settings/waivers` (The Firma.dev embed)
- `/dashboard/boats` (Adding fleet vessels)

---

## 2. The Guest View (Storefront / Onboarding)
Guests **do not create passwords** and **do not use the login screen**. They access the platform purely via private, cryptographic Trip Links generated by the Operator.

* **How to Verify:**
  1. Login as the **Operator**.
  2. Navigate to `/dashboard/trips` and create a new Quick Trip.
  3. The system will generate a **Trip Link** (e.g., `http://localhost:3000/trip/[slug]`).
  4. Open an **Incognito Browser Window** and paste that link.
* **Test Case Details**: You are now viewing the exact flow a Guest sees on their phone, including:
  - Entering their name and details
  - The Firma.dev e-signature widget (`StepWaiver.tsx`)
  - Add-on selection (Ice, Towels, etc.)

**Key Routes to Verify as Guest:**
- `/trip/[slug]` (The unified responsive onboarding wizard)

---

## 3. The Captain View (Operational Execution)
Similar to Guests, Captains **do not use the login screen**. They are given a highly secure "Snapshot Link" that remains active only for the duration of the trip. This link grants elevated privileges to modify trip state (starting/ending) but restricts overarching access to the Operator's business.

* **How to Verify:**
  1. Login as the **Operator**.
  2. Locate an active Trip in the Dashboard.
  3. Look for the "Share with Captain" or "Captain Link" button (this calls to `/api/captain/[token]`).
  4. Open that link in an **Incognito Browser Window**.
* **Test Case Details**: This interface is strictly locked to execution. You will test:
  - Verifying the guest manifest.
  - Hitting "Start Trip" (which automatically triggers Buoy insurance and USCG departure logs).
  - Hitting "End Trip".

**Key Routes to Verify as Captain:**
- `/captain/[token]` (The minimal mobile execution dashboard)
